# Kafka Transactions: Atomically Producing to Multiple Partitions

*Series: "Distributed Reliability: Kafka for Mission-Critical Pipelines" тАФ Post 2*
*Keyword: Kafka Transactional Producer*

---

In [Post 1](/posts/series3/3.1-idempotency), we explored the idempotent producer and how Kafka prevents duplicate messages within a single producer session on a single partition. The `enable.idempotence=true` configuration gives you exactly-once semantics *within a partition*. But what happens when your business logic requires writing to multiple partitions atomically тАФ where either all writes succeed or none do?

This is the problem Kafka Transactions were designed to solve.

## The Core Limitation: Idempotency Doesn't Cross Partitions

The idempotent producer guarantees that any message sent to a single partition will be written at most once, even if the producer retries due to a transient network failure or broker restart. However, this guarantee stops at the partition boundary.

Consider a price-publishing system that must atomically write a price update to two destinations: a `prices` topic for real-time consumption and a `price-audit` topic for regulatory compliance. Without transactions, you could successfully publish to `prices` and then fail before writing to `price-audit`. Your consumer sees the price; your auditor does not. In a regulated financial environment, this is an audit trail failure тАФ and a compliance incident.

Idempotency cannot solve this class of problem because it operates at the granularity of a single produce request to a single partition. Cross-partition atomicity requires a different mechanism.

## The Kafka Transactional API

Kafka introduces a formal transactional programming model built into the producer. The API follows a clear lifecycle:

```java
Properties props = new Properties();
props.put("bootstrap.servers", "kafka-broker-1:9092,kafka-broker-2:9092");
props.put("transactional.id", "price-publisher-001");
props.put("enable.idempotence", true);

KafkaProducer<String, PriceUpdate> producer = new KafkaProducer<>(props);

producer.initTransactions();

try {
    producer.beginTransaction();

    ProducerRecord<String, PriceUpdate> priceRecord =
        new ProducerRecord<>("prices", instrumentId, priceUpdate);
    ProducerRecord<String, PriceUpdate> auditRecord =
        new ProducerRecord<>("price-audit", instrumentId, priceUpdate);

    producer.send(priceRecord, (metadata, exception) -> {
        // Callback handles async completion
    });
    producer.send(auditRecord, (metadata, exception) -> {
        // Callback handles async completion
    });

    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
    throw e;
}
```

The four-method sequence тАФ `initTransactions()`, `beginTransaction()`, `send()`, and `commitTransaction()` тАФ forms the transactional boundary. `abortTransaction()` tears down the transaction on failure, discarding all messages written under that transaction's umbrella.

## Transaction Coordinator and Two-Phase Commit

Kafka achieves atomicity through a two-phase commit protocol managed by a component called the **Transaction Coordinator**, a logical role assumed of one broker in the cluster. This coordinator is not a separate service; it is a per-producer state machine running within an existing broker process.

When `initTransactions()` is called, the producer registers with the Transaction Coordinator and receives a `transactional.id` acknowledgement. The coordinator maintains transaction state in a private internal topic called **`__transaction_state`** (50 partitions, replicated like any topic). This topic stores the transaction's status тАФ `EMPTY`, `ONGOING`, `COMMITTING`, or `PREPARING` тАФ alongside the list of partitions involved.

The commit protocol proceeds in two phases:

1. **Phase 1 тАФ Prepare:** The producer calls `commitTransaction()`. The coordinator writes a `PREPARING` marker to `__transaction_state` and sends `PrepareCommit` requests to all brokers hosting the involved partitions. Each broker writes a special `PREPARED` marker to its local log but does not yet expose the messages to consumers.

2. **Phase 2 тАФ Commit:** Once all brokers acknowledge the prepare phase, the coordinator writes a `COMMIT` marker to `__transaction_state`. Brokers then expose the transaction's messages to consumers. If any broker fails to acknowledge, the coordinator issues an `Abort` on recovery, and the prepared messages are discarded.

This is the same pattern as a classic distributed transaction, but without an external XA coordinator. Kafka's log is the durable record of truth, replacing the two-phase commit's "transaction manager."

## Surviving Restarts with `transactional.id`

The `transactional.id` configuration is not merely a label тАФ it is a fencing mechanism that ensures no two producer sessions can hold the same transactional identity simultaneously.

When a producer registers with a `transactional.id`, the Transaction Coordinator records the assigned `producer_epoch` (a monotonically increasing number). If the same `transactional.id` reconnects after a JVM crash or network blip, the new session receives a *higher* `producer_epoch`. Any in-flight writes from the old (crashed) session are fenced off: brokers reject produce requests bearing a stale epoch, preventing the zombie producer from writing duplicate messages.

This is the critical property that makes Kafka Transactions suitable for leader election, consumer offset commits, and any workflow where a crashed process must not contaminate the pipeline after restart.

## Consumer Isolation: `read_committed` vs `read_uncommitted`

Transactions are only half the story. Consumers must opt into transactional semantics by setting `isolation.level` to `read_committed`.

With `read_uncommitted` (the default), consumers read all messages as they appear in the log, including those from aborted transactions that were never committed. The consumer has no visibility into whether a message was part of a successful transaction or a rolled-back one.

With `read_committed`, the consumer's consumer group coordinator holds back messages from incomplete or aborted transactions until the transaction is formally committed or aborted. The consumer sees only the final state: either the full transaction's messages or none of them.

For our price-pipeline use case, `read_committed` is non-negotiable. A consumer reading `prices` must not see a price update that was atomically paired with an audit entry that later aborted. The audit trail would be permanently inconsistent.

```java
Properties consumerProps = new Properties();
consumerProps.put("bootstrap.servers", "kafka-broker-1:9092");
consumerProps.put("group.id", "price-consumer-group");
consumerProps.put("isolation.level", "read_committed");

KafkaConsumer<String, PriceUpdate> consumer = new KafkaConsumer<>(consumerProps);
```

## The Latency Cost: Should You Use Transactions in a Real-Time Price Stream?

Transactions are not free. The two-phase commit protocol introduces additional round trips between the producer, the Transaction Coordinator, and the participating brokers. In a high-throughput price-publishing system where latency is measured in microseconds, this overhead is measurable.

A `commitTransaction()` call blocks until the coordinator confirms the commit. Under default configuration, this adds a round-trip to the coordinator on top of the usual broker acknowledgements. For a single-partition idempotent producer, `acks=1` produce can complete in under 1ms. A transactional produce to two partitions adds 2тАУ5ms of commit latency under moderate load.

Whether this is acceptable depends on your pipeline's SLAs. Regulatory audit systems often tolerate higher latency in exchange for strong consistency guarantees. Real-time price distribution to trading clients may need sub-millisecond latency, in which case you might accept at-least-once delivery to `prices` and use a separate, lower-throughput transactional pipeline for the audit topic.

A pragmatic architecture separates these concerns: the hot path (`prices`) uses idempotent producers with `acks=1` for speed, while the compliance path (`price-audit`) uses full transactions with `read_committed` consumers. The two pipelines serve different SLAs and different masters.

## Why This Matters in Production

Kafka Transactions are not a performance optimisation тАФ they are a correctness mechanism. In financial trading systems, audit trails are regulatory requirements. A single missing audit entry can trigger a fine. A duplicated entry can invalidate a back-test and cause a trading strategy to appear profitable when it was not.

Understanding the two-phase commit protocol, the role of `__transaction_state`, and the fencing semantics of `transactional.id` is what separates a candidate who knows Kafka APIs from one who understands Kafka's consistency model. At the Tech Lead level, you are expected to reason about these trade-offs and design systems that make conscious, documented choices between consistency and performance.

Transactions are the tool. Discipline in knowing when to use them тАФ and when not to тАФ is the craft.
