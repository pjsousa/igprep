# ZGC: Sub-Millisecond Pauses at Any Heap Size

In the previous post we explored Shenandoah GC — a concurrent collector that tackles the stop-the-world problem with Brooks pointer indirection. If Shenandoah is the mature workhorse of low-latency collectors, **ZGC (Z Garbage Collector)** is its more aggressive successor, purpose-built for workloads where even a 10ms pause is unacceptable. ZGC guarantees sub-millisecond pauses regardless of heap size, and it does so using a fundamentally different trick from Shenandoah: coloured pointers rather than indirection layers.

This post is Post 5 in the *JVM Internals: Taming Latency in Financial Systems* series. If you've been following the arc — from Survivor Spaces to TLABs to JIT compilation to Shenandoah — ZGC is the payoff: the collector that finally eliminates the GC pause as a latency concern, even at multi-terabyte scale.

---

## The Core Problem ZGC Solves

Every stop-the-world GC pause is a liability in a financial price-streaming system. When your p99 latency SLA is 5ms, a 50ms Young GC pause is a breach — and with G1GC, those pauses scale with the live data set size. At 16GB heap with 40% live data, you can see 20–80ms pauses regularly. Shenandoah reduced that to sub-10ms by doing compaction concurrently. ZGC takes this further: **pause times stay under 1ms regardless of heap size, even at multi-terabyte scale.**

This is the headline guarantee that makes ZGC compelling for trading platforms. A matching engine or risk calculator running with a 50GB heap doesn't want GC jitters. ZGC delivers that.

## Coloured Pointers: Encoding GC State in the Pointer Itself

Where Shenandoah uses an extra indirection pointer (the Brooks pointer) inside every object header to allow concurrent object movement, ZGC takes a more elegant approach: it encodes GC metadata **in the unused bits of a 64-bit pointer**.

A Java object reference on a 64-bit JVM has 48 bits of addressable space (the rest is reserved). ZGC uses three of those spare bits as *coloured pointers*:

- **Marked0** (M0) — object is live in the current GC cycle
- **Marked1** (M1) — object is live in the alternate GC cycle
- **Remapped** (R) — object reference points to the object's final location post-relocation

When you store a reference to an object, the JVM doesn't just store the memory address — it stores the address *plus* these flag bits. The CPU sees the full 64-bit value; the garbage collector tracks which colour the pointer currently has.

The key insight: **the pointer itself tells you whether the object has been moved**. No extra indirection needed. When ZGC relocates an object, it updates the coloured pointer to point to the new location and marks it `Remapped`. Any thread that later dereferences that pointer goes through a load barrier (more on that shortly) which catches stale colours and follows the forwarding pointer if needed.

This means the GC can move objects while threads run. Threads never block waiting for the GC to finish a relocation.

## Load Barriers: The Check at Every Dereference

Every time Java code dereferences an object — `priceTick.getInstrumentId()`, `order.getPrice()` — the JIT-compiled code runs a *load barrier*. This is a small check inserted by the JVM when it sees a pointer being loaded from memory:

```
if (pointer.marked_state != good_state) {
    // slow path: handle pending relocation
    slow_path(pointer);
}
```

In ZGC, the load barrier checks the coloured pointer's state. If the object has been relocated and the pointer hasn't been updated yet (or is in an intermediate colour), the barrier calls a helper that fixes the pointer — following the forwarding pointer if the object was moved.

This is why ZGC can keep application threads running uninterrupted: the load barrier handles the remapping *on the fly*, at the point of dereference, rather than requiring a stop-the-world pause to update all references to an object simultaneously.

The load barrier cost is measurable — typically 2–5% throughput overhead vs a stop-the-world collector. But in latency-sensitive systems, that trade-off is almost always worth it.

## Concurrent Relocation: The GC Phases

ZGC cycles through four main GC phases, most of which run *concurrently* with the application:

| Phase | Stop-the-World? | What Happens |
|---|---|---|
| **Pause Mark Start** | Yes (sub-ms) | Roots (thread stacks, registers) are scanned. Very brief. |
| **Concurrent Mark** | No | Traverses the heap graph, marking live objects. Coloured pointers encode mark state. |
| **Pause Mark End** | Yes (sub-ms) | Marks the end of marking, transitions colour states. |
| **Concurrent Relocate** | No | Objects are moved to new locations; forwarding pointers are installed. |
| **Concurrent Remap** | No | Old references are updated to point to new locations; load barriers also handle this lazily. |

The three stop-the-world pauses — Mark Start, Mark End, and Relocate Start — are all designed to be *sub-millisecond*. They involve scanning root registers and thread stacks, which is a small, bounded amount of work regardless of heap size. This is why ZGC's pause guarantee doesn't degrade with scale.

## Generational ZGC: The JDK 21+ Improvement

Up until JDK 20, ZGC was a *single-generation* collector — it managed a flat heap. This meant it couldn't leverage the generational hypothesis (most objects die young), which increased throughput cost vs generational collectors like G1.

**Generational ZGC** (enabled by default in JDK 21) adds a young generation (Eden + Survivors) alongside ZGC's mature generation. The collector now runs two-colour cycles independently:

- **Young GC**: Minor collections targeting Eden — very fast, leverages the generational hypothesis
- **Concurrent GC**: The mature generation is collected using the same coloured-pointer concurrent algorithm

The result is that Generational ZGC delivers **throughput close to G1** (often 90–95% of G1) while maintaining ZGC's signature sub-1ms pause times. For fintech workloads that need both throughput *and* latency guarantees, this is the best of both worlds.

```bash
# Enable Generational ZGC (JDK 21+)
-XX:+UseZGC -XX:+ZGenerational

# Classic (non-generational) ZGC
-XX:+UseZGC -XX:-ZGenerational
```

## Shenandoah vs ZGC: The Indirection Trade-off

It's worth pausing to contrast ZGC's coloured-pointer approach with Shenandoah's Brooks pointer, because interviewers will sometimes ask "why not Shenandoah?" or "how does ZGC avoid the overhead Shenandoah has?"

**Shenandoah** inserts a forwarding pointer *inside every object header*. When an object is moved, the old location becomes a forwarding object whose header points to the new location. Every dereference goes through this indirection — even in the fast path, there's an extra memory load. Shenandoah's pause times are sub-10ms, but the Brooks pointer adds measurable per-dereference overhead.

**ZGC** encodes state in the *pointer*, not the object. An object header in ZGC is unchanged from a normal JVM object — no extra field, no extra memory load on dereference in the common case (when the pointer is already correctly coloured). The load barrier cost is there, but it's generally lower than Shenandoah's indirection overhead for read-heavy workloads.

The other key difference: ZGC's coloured pointers require **64-bit architecture with compressed pointers disabled or adjusted** — the spare bits only exist in a 64-bit address space. Shenandoah works on both 32-bit and 64-bit. Shenandoah also has a larger JVM engineering footprint (the Brooks pointer touched every object header), while ZGC's coloured pointers are primarily a runtime/c1 stub concern.

Both are production-grade. ZGC tends to win on raw latency consistency; Shenandoah wins in environments where memory overhead matters or where the team is already running OpenJDK with Shenandoah backports.

## Key JVM Flags

```bash
# Basic ZGC启用
-XX:+UseZGC

# Generational ZGC (JDK 21+, recommended for throughput)
-XX:+UseZGC -XX:+ZGenerational

# Soft Max Heap Size — ZGC will try to keep heap below this
# without triggering aggressive GC; useful for multi-tenant environments
-XX:SoftMaxHeapSize=32G

# Set max heap; ZGC works well with large heaps
-Xmx64G

# Verbose logging
-Xlog:gc*:file=gc.log:time,uptime,level,tags

# ZGC-specific diagnostic flags
-XX:+VerboseGC    # Equivalent to above in most cases
```

## When to Reach for ZGC

ZGC is the right choice when:

- **p99 latency SLA is sub-10ms** and GC pauses are the primary risk to that SLA
- **Heap size is large** (16GB+): ZGC's pause guarantee is independent of heap size, making it uniquely suited to big heaps
- **Application is throughput-tolerant**: ZGC's concurrent phases consume CPU; budget 10–15% more CPU than a stop-the-world collector
- **You're on JDK 21+**: Generational ZGC resolves the earlier throughput disadvantage

ZGC is probably *not* the right choice when:

- **CPU is the bottleneck**: if you're already at 80%+ CPU and can't spare the headroom for concurrent GC phases
- **Memory is heavily constrained**: ZGC maintainsColourful metadata and concurrent structures that add some overhead
- **p99 latency in the 10–50ms range is acceptable**: G1GC or Shenandoah may be simpler and offer better throughput

For a price-streaming service at a trading firm, ZGC is almost always worth seriously considering. A 64GB heap with Generational ZGC will deliver p99 under 1ms pauses while maintaining >90% throughput — a combination G1GC simply cannot achieve under load.

## Why This Matters in Production

In a live trading system, a 100ms GC pause isn't just a latency spike — it can be a regulatory event. If your price-delivery pipeline stalls for 100ms during market open, downstream clients may see stale prices and make incorrect trading decisions. The risk engine, if it misses a heartbeat from the price cache, may conservatively halt trading entirely.

ZGC eliminates GC as a category of operational risk. With `-XX:+UseZGC -XX:+ZGenerational` and a properly sized heap, you can run a JVM at 50GB+ heap and know that *no GC pause will breach your latency SLA*. That certainty — provable with an HDRHistogram at runtime — is what separates a production-grade low-latency stack from one that's just been tuned with heuristics.

The operational discipline this enables: GC tuning becomes a JVM flag decision rather than a quarterly investigation. You set the flags, you run the benchmark, you deploy. The collector does the rest.
